Artifact Descriptions

This file contains extracted html files from Coq files. Since the current Coq files have many dependencies with CompCert and CertiKOS, we submit these extracted files instead of bare Coq files.

Machine models and refinement proofs

  • Concurrent_Linking_Lib.v: This contains multiple auxiliary definitions that are required in our language definitions and proofs.
  • Concurrent_Linking_Def.v: This contains all intermediate language definitions.
    • HardWaredSetting: It provides abstract state definitions.
    • command: It provides abstract commands.
    • HardSemantics: It provides abstract hardware semantics.
    • hardware_local_step: It provides local step rules with abstract definitions.
    • hardware_step: It provides a formal definition of our SimplMM
    • env_step: It provides an environment machine model. Differently with our submission, it contains one more intermediate language, "oracle_step", which is quite similar to the environment machine model with the full core set.
    • single_step: It provides a single core machine model.
  • Concurrent_Linking_Additional_Prop.v: This file contains several proofs that are not directly used in the connection between SimplMM and LAsm.
    • one_step_oracle_refines_hardware: It shows the Theorem 2 in Section 6.
  • Concurrent_Linking_Prop.v: Refinement proofs between machine models that are defined in Concurrent_Linking_Def.v.
    • ORACLE_REFINES_HARDWARE: This module provides the refinement theorem that corresponds to Theorem 4. As we have mentioned, we introduce one more abstract machine mode, "oracle_step". Therefore, this theorem actually uses "oracle_step" instead of "env_step".
    • SINGLE_REFINES_ENV This provides a theorem corresponding to Theorem 3 in our submission.

    Linking with CCAL

    Linking with CCAL is described in Section 7.1 and 7.2. There are some steps that are required to use our abstract definitions and proofs with CertiKOS on CCAL. In addition to that, there are some extra works to provide concrete machine models and refinement proofs that are compatible with CompCert's proof library. Even though the size of those definitions are not small, they are quite systematic, so we could define them quite quickly.

    • Instantiate hardware setting (HardWareSetting) and program transition rules (HardSemantics) that are required to instantiate all machine models in Concurrent_Linking_Def.v. This is the part that is described in Section 7.1 and 7.2.
      • HWSemImpl.v: This provides concrete definitions for our abstractions.
    • Instantiate all the machine models by using the instances of HardWareSetting and HardSemantics.
      • HWStepSemImpl.v: This provides a concrete machine model of "hardware_step".
      • EnvSemImpl.v: This provides a concrete machine model of "env_step".
      • SingleSemImpl.v: This provides concrete definitions of "single_step".
    • Instantiate all the refinement proofs with those concrete machine models
      • AsmOracleSemtoHWStepSem.v: This provides concrete definitions of Theorem 4, by using a CompCert's style.
      • AsmSingleSemtoEnvSem.v: This provides concrete definitions of Theorem 3, by using a CompCert's style.

    Top level theorem

    • Concurrent_Linking: This file provides the top level theorem of our work, which corresponds to Theorem 7 in the submission (Section 7.3)
